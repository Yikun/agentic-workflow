### 1. Overview
本项目目标是构建一个基于 GitHub Issue 的 Agentic Workflow，用于自动化完成需求分析、架构设计与代码开发、测试执行三个阶段的端到端流程。流程应以单个 Issue 为主线，产出可审阅的文档与交付物，并在满足阶段条件后推进到下一阶段。

### 2. Functional requirements (FR-01, FR-02, ...)
FR-01：系统应支持从指定 GitHub Issue 读取原始需求描述作为流程输入。

FR-02：系统应在需求分析阶段输出结构化需求文档，并由 requirements 与 requirements-qa 两类 Agent 分别完成初稿与质量校验。

FR-03：系统应将需求分析阶段结果发布为该 Issue 的评论，供用户审阅。

FR-04：系统应仅在原 Issue 作者发布的评论满足以下全部条件时，进入第二阶段：（1）评论正文去除首尾空白后精确等于字符串 "/approve"（大小写敏感，不允许包含其他任何文本）；（2）该评论未被编辑且未被删除；（3）若存在多条满足条件的评论，以最早发布的一条为准。反应、混合文本评论或他人评论均不构成批准；若该批准评论被编辑或删除，批准立即失效。

FR-05：系统应在第二阶段基于已批准的需求文档产出系统架构设计文档。

FR-06：系统应在第二阶段基于已批准的需求文档与架构设计产出代码变更，所有代码文件须存储至当前仓库的 `artifacts/src/` 目录下。

FR-07：系统应在第二阶段完成后创建 Pull Request（PR）用于提交代码变更。

FR-08：系统应仅在该 PR 当前 HEAD 提交上的全部"Required"状态检查均为成功（success）时，判定 CI 通过，并在原 Issue 评论中触发并声明自动进入第三阶段；只要任一 Required 检查为失败、取消、超时、待运行或缺失，均不得进入第三阶段；若 PR 上未配置任何 Required 检查（空集合），视为 CI 未通过，不得进入第三阶段。

FR-09：系统应在第三阶段由 tester Agent 逐条读取 artifacts/03-test-cases.md 中定义的验收测试用例，针对 artifacts/src/ 中的交付代码逐条模拟执行（通过调用代码、检查输出或逻辑推演方式验证），并将每条用例的执行结果（通过/失败/跳过及原因）汇总写入 artifacts/04-report.md；报告须包含：结果汇总表（总数/通过/失败/跳过）、逐条结果说明、失败详情分析、最终 PASS 或 FAIL 结论；FR-09 所指验收测试与 FR-08 中 CI 自动化测试属于不同集合，不得相互替代。

FR-10：系统应为每个阶段产出明确的阶段性结果（需求文档、架构与代码交付、测试报告），并可由用户在对应 Issue/PR 上追踪。

FR-11：系统对外沟通内容与产出文档应使用中文。

FR-12：系统应将本项目所有阶段的自动化触发逻辑定义为 GitHub Actions Workflow 文件，存储于当前仓库的 .github/workflows/ 目录下，格式为 YAML；每个阶段或子阶段至少对应一个独立的 Workflow 文件，覆盖需求分析、架构设计、编码开发、测试执行全部阶段的自动化触发与阶段门禁逻辑。

### 3. Non-functional requirements
NFR-01：流程可追溯性：每个阶段的输入、输出与状态变更应在 GitHub Issue/PR 上可审计。

NFR-02：阶段门禁可靠性：阶段切换条件必须严格满足，禁止越过审批或 CI 状态的提前推进。

NFR-03：结果一致性：同一输入条件下，结构化文档的章节结构与编号规则应保持一致。

NFR-04：可读性：需求、架构、测试相关文档应结构清晰、术语一致、便于人工审阅。

NFR-05：自动化连续性：在外部依赖可用的前提下，除各阶段明确规定的人工审批操作（如 FR-04 的 /approve 评论）外，阶段间衔接不应要求用户进行任何额外手动操作。

### 4. Assumptions
ASS-01：仓库已具备必要的 GitHub 权限配置，可读取 Issue、发布评论、创建 PR 并读取 CI 状态。

ASS-02：DASHSCOPE_API_KEY 已正确配置且可用于相关 Agent 执行。

ASS-03：工作流运行环境可使用 opencode GLM-5（Alibaba China）作为模型能力来源。

ASS-04：用户会在需求分析阶段对 Issue 评论中的结果进行明确批准或驳回。

ASS-05：项目已提供或可提供第三阶段所需测试用例输入。

### 5. Out of scope
OOS-01：不包含对 GitHub 平台能力本身的改造。

OOS-02：不包含与本工作流无关的业务系统功能开发。

OOS-03：不包含人工编写的长期运维流程与组织治理流程定义。

OOS-04：不包含模型供应商账号、计费或网络环境问题的解决。
